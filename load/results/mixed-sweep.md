# 부하 테스트 결과 — 종합 트래픽 믹스 동시성 스윕

> ⚠️ **측정 시점 주의**: 이 결과는 2026-07-08의 MVC+가상스레드+JPA 전환 **이전**, 즉 구 리액티브 스택
> (WebFlux + 코루틴 + R2DBC)에서 측정된 기록입니다. 현재 스택의 수치가 아니라 당시 기록으로 보존합니다.

## 테스트 개요

- **대상**: 게시판 API (Spring WebFlux + Kotlin 코루틴 + R2DBC + Redis + Elasticsearch)
- **시나리오**: `load/scenarios/mixed.js` — 읽기 70% : 쓰기 20% : 검색 10% 혼합 트래픽
- **방식**: k6 `ramping-arrival-rate` — 목표 RPS(초당 요청)를 500 → 8000까지 계단식으로 올리며
  시스템이 무너지는 지점(knee)을 탐색
- **각 레벨 프로파일**: 워밍업 5s + 램프 5s + **지속 30s** + 램프다운 3s
- **환경**: 로컬 단일 머신(macOS) — 앱 + PostgreSQL + Redis + Elasticsearch + k6가 **같은 CPU 공유**
  (아래 "한계와 해석" 참고)

| 500 | 17,863 | 0 | 0% | 4.83 | 7.16 | 36.2 | 2.52 | 5.62 | 7.14 | ✅ |
| 1,000 | 35,675 | 0 | 0% | 3.23 | 5.32 | 24.4 | 1.68 | 4.01 | 4.81 | ✅ |

> "드롭"은 부하 생성기가 목표 도착률을 맞추지 못해 버린 요청 수 = 시스템이 그 속도를 못 따라갔다는
> 신호. 4,000까지 드롭 0 → 목표 RPS를 완전히 소화. 8,000에서 25,725건 드롭 + 지연 폭증 → 포화.

### 리소스 사용 (부하 중 Actuator 메트릭)

- **R2DBC 커넥션 풀 최대치: 10개** — 4,000 RPS(30초에 14.2만 요청)를 커넥션 단 10개로 소화
- **JVM 스레드 피크: 70개** (이벤트 루프 + GC 등 포함) — 요청당 스레드를 쓰지 않는 논블로킹 구조의 증거

## 한계와 해석 (정직하게)

- **4,000 RPS까지 선형 확장**: 부하가 8배(500→4000) 늘어도 p99는 7ms → 10ms로 거의 일정. 에러 0%.
- **knee(포화점)는 4,000~8,000 RPS 사이**: 8,000에서 처음으로 드롭(25,725건)과 지연 폭증(p99 2.4s) 발생.
- **포화해도 에러율 0%**: 8,000에서도 5xx/타임아웃 에러는 없었고, 초과분은 큐잉(지연 증가)되거나 부하
  생성기가 버렸을 뿐 — **붕괴(collapse)가 아니라 완만한 성능 저하(graceful degradation)**.
- **⚠️ 환경 보정**: 로컬 단일 머신에서 k6·앱·PostgreSQL·Redis·ES가 **같은 CPU를 나눠 씀**. 8,000 RPS
  포화는 앱 단독 한계가 아니라 머신 전체 포화에 가깝다. 실제 배포(앱/DB/Redis/ES 분리 + 부하 생성기
  별도 노드)라면 knee는 더 높아진다. 따라서 수치는 **"단일 개발 머신 기준"**으로 해석해야 한다.

## 이 결과가 증명하는 설계 강점

1. **논블로킹 스택의 고동시성 효율** — 커넥션 10개 / 스레드 70개로 4,000 RPS·p99 10ms 유지.
   요청당 스레드를 쓰는 블로킹 스택이라면 수천 동시 요청에서 스레드가 고갈됐을 지점.
2. **좁은 트랜잭션 경계 + 외부 I/O 격리** — 부하 중 커넥션 풀 고갈·대기(pending) 없이 안정.
3. **Redis 조회수 버퍼링의 쓰기 절감** — 4,000 RPS의 70%가 읽기 ≈ 초당 2,800 조회. 순진한 구현이면
   초당 2,800건의 조회수 UPDATE가 인기글 row에 몰린다. 이 프로젝트는 Redis `HINCRBY`로 모은 뒤
   1,000건 단위 배치 write-back → **조회수 DB 쓰기를 약 1,000:1로 압축**(락 경합 제거).

## 포트폴리오용 문장 (그대로 사용 가능)

- 리액티브(WebFlux/코루틴/R2DBC) 게시판 API에 k6 부하 스윕을 적용해 **단일 개발 머신에서 4,000 RPS를
  p99 10ms·에러율 0%로 처리**함을 검증. DB 커넥션 10개·JVM 스레드 70개만으로 달성해 논블로킹 구조의
  동시성 효율을 정량 입증.
- 부하를 8배(500→4,000 RPS)로 올려도 p99 지연이 7→10ms로 거의 일정한 **선형 확장**을 확인하고,
  포화점(knee)을 4,000~8,000 RPS 구간으로 식별. 포화 상태에서도 에러 0%로 **완만한 성능 저하** 확인.
- 조회수를 Redis에 버퍼링 후 배치 write-back하는 설계로 **인기글 조회수 DB 쓰기를 약 1,000:1로 절감**,
  읽기 폭주 시 DB 쓰기 경합 제거.
