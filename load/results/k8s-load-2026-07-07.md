# 부하테스트 결과 — k8s(Helm) 배포 · 2026-07-07

> 기존 `mixed-sweep.md` / `large-scale.md`는 **IntelliJ 단독 실행 + 코드 변경 이전** 측정치다.
> 이 문서는 **현재 코드**를 **k8s(kind+Helm)** 에 배포한 상태에서 다시 측정한 결과다.

## 환경

| 항목 | 값 |
|---|---|
| 실행 형태 | kind(단일 노드) + Helm, **관측성(LGTM) ON** |
| colima | **4 vCPU / 8 GB** (앱2 + PostgreSQL/Redis/Elasticsearch/Kafka + Grafana/Mimir/Loki/Tempo/Alloy 를 전부 공유) |
| board-service | requests `cpu=250m mem=512Mi`, limits `mem=1Gi` (**CPU 하드 limit 없음**, 공유 코어 경합) |
| 시드 데이터 | `board` **100,000행**, `users` 100명(실 BCrypt), ES `boards` 색인 동기화됨 |
| 부하 도구 | k6 v2.1.0 (호스트 macOS에서 실행), 대상 `http://localhost:8080`(NodePort 30080) |
| 릴레이 | `board.outbox.relay.enabled=true`, poll 1s / batch 100 |

**해석 주의**: k6·앱·모든 인프라·관측성 스택이 **4코어/8GB 한 대**를 나눠 쓴다. 따라서 **절대 처리량(RPS)은 IntelliJ 단독 때보다 크게 낮다** — 이는 성능 저하가 아니라 자원 분할이다. 깊이무관성·선형성·포화 거동·병목 위치 같은 **정성적 결론은 유효**하다.

---

## 1) 키셋 페이지네이션 깊이무관성 — `pagination.js`

100,000행, PEAK_RATE=1500, 65s. 커서 깊이를 shallow/mid/deep(=전체 95% 지점, OFFSET ~95,000 등가)로 균등 분산해 depth 태그별 p95 비교.

| 깊이 | OFFSET 등가 | p95 | p99 |
|---|---:|---:|---:|
| shallow | ~0 | **7.42 ms** | 57.4 ms |
| mid | ~47,500 | **7.52 ms** | 61.5 ms |
| deep | ~95,000 | **7.70 ms** | 63.1 ms |

- 세 깊이 p95 편차 **0.28 ms** → 깊이 무관 평탄(인덱스 시크). OFFSET이었다면 deep이 수십~수백 배 느려야 함.
- 76,124 요청, **에러율 0%**, threshold(`p95<50ms`) 3버킷 모두 통과.

✅ 대용량에서도 페이지 깊이와 무관하게 일정 지연. 코드 변경 후에도 재현됨.

## 2) 종합 트래픽 스윕(읽기70:쓰기20:검색10) — `mixed.js`

각 레벨 warmup+ramp+sustain. Redis 조회수 증가 / DB write+아웃박스 기록 / ES 검색이 동시 경합.

| 목표 RPS | 유효 처리량 | 에러율 | read p95 | write p95 | search p95 | 판정 |
|---:|---:|---:|---:|---:|---:|:--:|
| 1,000 | ~739/s | 0% | 9.07 ms | 11.9 ms | 15.8 ms | ✅ |
| **1,500** | **~1,153/s** | **0%** | 16.6 ms | 18.3 ms | 18.1 ms | ✅ |
| 2,000 | 566/s ↓ | 5.78% | **27.8 s** | **27.9 s** | 8.6 s | ❌ 붕괴 |
| 3,000 | 377/s ↓ | 12.2% | 35.1 s | 35.3 s | 10.0 s | ❌ 붕괴 |

- **knee = 1,500 ~ 2,000 RPS 사이.** 지속 가능 상한 ≈ **1,150 req/s(유효)**.
- knee를 넘으면 처리량이 오히려 **감소**(1153 → 566 → 377) — 전형적 혼잡 붕괴(congestion collapse), 대기열 폭증.
- **병목 위치가 뒤바뀜**: 포화 시 `read`/`write`(board-service+PostgreSQL 경로) p95가 ~28초로 먼저 무너지는데 `search`(ES, 별도 파드)는 8.6초로 상대적으로 버팀.
  - 이전(IntelliJ) 결론은 "ES가 1차 병목"이었으나, **여기선 board-service/PG가 1차 병목** — CPU limit 없이도 4코어를 obs·datastore와 공유해 앱 CPU가 먼저 고갈되기 때문. 즉 **병목은 코드가 아니라 배포 자원 프로파일**이 결정.

## 3) 이벤트 파이프라인 — 아웃박스 → Kafka → search-indexer → ES

PEAK_RATE=1000 쓰기 부하(≈16,300 writes) 기준 실측. board-service 쓰기는 DB+아웃박스를 원자적으로 남기고, 릴레이가 Kafka로 발행 → search-indexer가 ES 색인.

| 지표 | 관측 |
|---|---|
| `board.indexer.indexed` Δ | 2 → 8,783 (부하 중), 이후 계속 증가 |
| `board.outbox.unpublished`(백로그) | 부하 종료 시 **7,380** → 이후 단조 감소(자가 드레인) |
| 드레인 거동 | 7,380 → 980 (80초), **유실 0** |
| `board.indexer.dlq` | **0** (전 구간, 실패 격리 없음) |
| ES bulk write(`board.indexer.batch`) | 평균 ~2.8 ms, 최대 29 ms |
| ES `boards` 문서 수 | 100,001 → 108,782 (색인 델타와 일치) |

- **최종 일관성 실증**: 쓰기 폭주로 아웃박스 백로그가 쌓여도 릴레이가 뒤따라 **손실 없이 드레인**. DLQ 0.
- **릴레이 처리량이 튜닝 포인트**: batch=100 @ poll 1s → 상한 ~100 events/s. 부하 시 쓰기 유입(~150/s)이 이를 초과해 백로그 축적. `board.outbox.unpublished` 게이지가 이 지연을 정확히 포착.

## 발견 & 개선 제안

1. **knee가 낮은 건 자원 때문**: 4코어/8GB에 obs까지 얹어 매우 빡빡. 순수 최대치 측정 시 `up.sh`(obs 없이) + colima 12G로 올리면 knee가 크게 상승. 관측성 관찰과 최대 처리량 측정은 트레이드오프.
2. **board-service에 CPU request/limit 명시** + replica 2로 스케일하면 knee 개선 여지.
3. **릴레이 배치 상향**(batch 100→500, poll 1s→200ms)으로 파이프라인 처리량↑ → 백로그 축적 완화.
4. 포화 시 board-service/PG가 먼저 무너지므로, 실운영에선 **R2DBC 풀·앱 CPU가 1차 스케일 대상**.

## 개선 2차 적용 (2026-07-07 오후) — 4가지 개선 + 발견

앞의 결과를 바탕으로 개선을 시도했다. **replicas 증가(및 그 안전화 락 코드)는 최종적으로 되돌렸고**, 나머지(릴레이 튜닝·풀·자원)만 적용했다.

### 적용한 변경 (커밋 대상)

| # | 항목 | 상태 | 근거 |
|---|---|---|---|
| 3 | **릴레이 처리량 튜닝** | ✅ 적용·실증 | batch 100→**500**, poll 1s→**200ms** (상한 ~100/s → **~2500/s**). 실측: 쌓여있던 **아웃박스 백로그 ~20,000건이 즉시 0으로 드레인**(이전엔 ~100/s로 수 분) |
| 4 | **R2DBC 풀 + postgres 정합** | ✅ 적용 | 풀 max-size 30→**50**. postgres `max_connections` 기본 100→**200**(풀+admin/reindex/스키마초기화 동시 커넥션 여유). **replicas × pool ≤ max_connections** 제약을 이 과정에서 확인 |
| 1 | **자원 상향/obs 토글** | ✅ 적용 | colima 8G→**12G**. obs on/off를 `--set observability.enabled` 로 토글해 최대처리량 측정 구성 확보 |
| 2 | **CPU request/limit 명시** | ✅ 적용 | board-service `requests cpu=500m`, `limits cpu=3000m` |
| 2 | **replicas 2로 증가** | ❌ **되돌림(replicas=1 유지)** | 단일노드 kind(4코어) VM이 board-service JVM 2개를 부하에서 감당 못 함(아래). 이를 위해 도입했던 릴레이·아카이브 분산 락 코드도 **함께 전부 원복** |

### replicas 증가를 되돌린 이유 — 환경 한계 (정직한 기록)

replicas=2로 knee 상승을 측정하려 했으나, **이 단일노드 kind(colima 4 vCPU) 환경은 board-service JVM 2개를 부하 상태에서 감당하지 못했다.** 1,500 RPS에서 매번 ~1,000요청 후 컨트롤플레인(kube-apiserver)이 다운됐고, obs on/off·8G/12G·CPU limit·max_connections와 **무관하게 재현**됐다:

- 원인: 2개 JVM의 2배 GC/netty/스레드 오버헤드가 4코어를 포화 → kubelet/apiserver 굶주림. 추가로 **k6(호스트)와 colima VM(vCPU)이 물리 CPU를 두고 경합**해, 반복 재시작 후에는 replicas=1·1,500 RPS도 불안정해짐.
- 새 코드 결함이 아니라 **배포 자원/하이퍼바이저 한계**(기존 메모의 "colima 고부하 불안정"과 일치).

**결론**: 로컬은 **replicas=1**로 운영한다. 멀티 레플리카가 필요하면 (a) 아웃박스 릴레이·아카이브에 분산 락(중복 실행 방지)을 먼저 넣고, (b) k6를 별도 머신 또는 앱 전용 노드가 있는 실 클러스터에서 측정해야 한다. (이번에 락은 시도했다가 요구에 따라 전부 원복했다.)

### 검증된 것 / 미검증된 것 요약

- ✅ 코드 게이트: ktlint + 전체 test(Konsist·Testcontainers) + koverVerify ≥92% 통과
- ✅ 릴레이 튜닝: 백로그 20k→0 즉시 드레인 실증
- ✅ 앱 기능: replicas=1·obs off 배포 후 health/list 200, 데이터 148k행 보존
- ⚠️ 미완: replicas=2 하에서의 knee 상승, obs-off 최대처리량 knee (환경 한계로 중단)
- ❌ 되돌림: 다중 레플리카(replicas=2)와 그 안전화 락 코드
